# 验收期真相对齐（物理诊断）

## 何时使用

满足任一即切换到此剧本（见 `CLAUDE.md` §验收期真相对齐）：

- 用户反馈「做的和说的不一样」、验收项与预期不符
- `memory/ROADMAP.md` 某项验收失败
- `scripts/check-doc-drift.sh` 报漂移（机检不变量）

## 漂移类型（诊断后标注）

| 类型 | 含义 |
|------|------|
| **实现漂移** | 文档/任务对，代码漏写或写错 |
| **理解漂移** | 代码自洽，但与业务直觉或决策不符 |
| **定义漂移** | 初始需求模糊，需先补 spec 再改代码 |

---

## 物理诊断四步（未完成前禁止改业务代码）

将 `<功能名称>`、`<具体现象>`、`<测试命令>` 替换为实际值后执行：

### 1. 强溯源

读取以下原文（按相关性选读，**不以** `proposal.md` 为权威）：

- `CLAUDE.md` — 相关约束与铁律
- `specs/core/tasks.md` — 对应 M0–M4 任务与 AC
- `memory/DECISIONS.md` — 如有相关已记录决策

输出：摘录与现象相关的**原文条款**（带文件路径）。

### 2. 差异比对

- `git diff` 或读取 `apps/desktop/src/`、`services/orchestrator/src/` 相关源码
- 列出：**文档/任务要求** vs **当前实现** 的具体逻辑偏差（精确到文件与行为）

### 3. 验证假设

- 跑 `./scripts/verify.sh`，或针对性 `cd services/orchestrator && uv run pytest <文件>`
- 读取终端日志；有现有测试则必须运行并引用结果

### 4. 输出诊断报告

```markdown
## 诊断报告 — <功能名称>

- **漂移类型**：（实现 / 理解 / 定义）
- **证据链**：
  - 文档原文：…
  - 当前代码：…（路径 + 行为）
  - 测试/构建结果：…（终端摘要）
- **修复方案**：需修改的精确文件列表与逻辑补充
```

## 修复路径（诊断确认后）

### A 类 · 实现漂移（文档对，代码错）

1. 修正代码逻辑
2. 增加防回退测试用例
3. 运行 `./scripts/verify.sh`，附终端日志
4. 在 `memory/FAILURES.md` 记录并标记防线

### B 类 · 定义漂移（文档模糊或理解偏差）

1. 补齐 Spec：更新 `specs/core/tasks.md` 或 `memory/DECISIONS.md`（架构级）
2. 验证先行：先写 FAIL 测试，再改代码至通过
3. 证据链：代码 diff + 终端日志
